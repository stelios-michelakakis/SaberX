# EDF SABER Platform Features

EDF SABER is a database-native engineering document management platform for structured, interdependent workbook-like artifacts such as CONOPS, ICD, and RTM. It replaces separate Excel workbooks with a unified repository designed for traceability, consistency, and controlled schema evolution.

## Document Repository

- Create and manage multiple engineering documents in one system.
- Store document metadata including title, description, status, classification, owner, baseline state, provenance, and audit context.
- Maintain Excel as an import and export format, while PostgreSQL remains the system of record.
- Support generic document structures so new artifact families can be added beyond CONOPS, ICD, and RTM.

## Reserved Document Structure

- Every document is created with three permanent reserved sections: `INSTRUCTIONS`, `GLOSSARY`, and `OPEN ISSUES`.
- Reserved sections are system-controlled and always remain in the required order.
- `INSTRUCTIONS` stores document-specific guidance and is editable by authorized users.
- `GLOSSARY` is generated by the system from document, schema, field, enum, and ID-policy metadata.
- `OPEN ISSUES` provides built-in issue, question, and decision tracking with standardized fields.

## Schema-Driven Data Management

- Documents contain structured sheets with metadata-defined fields.
- Each field stores a label, machine slug, type, description, validation settings, required flag, uniqueness flag, editability, visibility, and ordering.
- Supported field types include auto ID, text, numbers, boolean, dates, enums, references, URLs, statuses, tags, and rich notes.
- Field descriptions are first-class metadata and feed glossary generation, field legends, help context, and search indexing.

## Stable Identity And Visible IDs

- Every stored row uses an immutable UUID as its true database identity.
- Visible engineering IDs are generated for human use and are never used as primary keys.
- References are stored using UUIDs, so renumbering visible IDs does not break traceability.
- ID policies support sheet-level prefixes and configurable zero padding.
- Compound multi-prefix IDs are intentionally disallowed.

## Trace Links And References

- Rows can reference rows from the same document or, in later phases, other documents.
- Single-reference and multi-reference fields store immutable row UUID links.
- Reference rules can constrain which sheets and ID-bearing rows are valid targets.
- Deletion of referenced rows is blocked by default to prevent silent data corruption.

## Consistency And Impact Control

- Destructive or schema-changing operations can be evaluated before they are applied.
- Impact analysis identifies affected data, references, derived records, and blockers.
- Integrity checks detect broken links, invalid typed values, missing required values, duplicate visible IDs, orphaned records, and schema-related inconsistencies.
- Open integrity issues are stored for review and resolution.

## Audit Logging And Traceability

- All meaningful actions are recorded in an immutable audit log.
- Audit events include actor, timestamp, action type, entity type, entity identifier, before state, after state, diff data, source type, success status, and transaction correlation.
- Logged actions include document changes, sheet changes, schema changes, row and cell changes, reference updates, imports, exports, snapshots, permission changes, security events, and system-generated updates.
- Deletions preserve reconstructable historical state through soft deletion, tombstones, and serialized pre-delete data.

## Snapshots, Baselines, And Diff

- Documents can be captured as named snapshots.
- Baseline states support lifecycle tracking such as seed, draft, under review, baselined, and superseded.
- Snapshot comparison identifies added, removed, and changed entities.
- Snapshots are also audit-tracked.

## Excel Import

- `.xlsx` workbooks can be imported as new documents.
- Import records preserve filename, file hash, importing user, timestamp, job status, and summary data.
- Workbook sheets are parsed into structured sheets and fields.
- Field types are inferred from source values and can be refined after import.
- Reserved sections are reconciled or created so imported documents still follow platform rules.

## Excel Export

- Documents can be exported back to `.xlsx` workbooks.
- Reserved sections are always included.
- Field labels, visible IDs, row values, references, sheet descriptions, and field descriptions are preserved in a deterministic logical format.
- Export jobs are recorded with status, requester, generated filename, summary data, and audit events.

## Search And Analysis

- Search covers document metadata, sheet metadata, field labels, field descriptions, row values, glossary content, and open issue content.
- Results are indexed in PostgreSQL using full-text search and trigram matching.
- Search entries retain document, sheet, field, row, match type, excerpt, and relation context.
- Relation-aware expansion can surface linked records alongside direct matches.

## Authentication And User Management

- The platform has no public registration.
- Initial access uses a one-time bootstrap administrator credential that must be changed immediately after first login.
- Non-bootstrap users are onboarded only through administrator-created invitations.
- Invitations are bound to email, single-use, revocable, valid for 48 hours, and stored using token hashes rather than plaintext tokens.
- Users have unique usernames and email addresses.
- Passwords are hashed with Argon2id.
- Users can manage basic profile information and change their own passwords.
- Administrators can manage invitations, users, roles, account status, and forced password-change requirements.

## Access Control And Security

- Role-based authorization supports administrator, document manager, editor, and reviewer/read-only responsibilities.
- Permission scopes can apply at application, document, and later sheet levels.
- Permission changes are auditable.
- Sessions are server-side and use secure HTTP-only cookies.
- Production deployment is designed for HTTPS-only operation behind a reverse proxy.
- Login attempts, failed logins, password changes, invitations, account activation, account disablement, and account archival are audit-tracked.

## Deployment And Operations

- The application is structured as a Next.js and TypeScript monolith backed by PostgreSQL.
- Drizzle manages the relational schema and migrations.
- Background processing supports derived-data refresh, search indexing, integrity checks, imports, exports, and snapshots.
- Docker Compose provides local services for application, worker, and database processes.
- The architecture is suitable for personal hosting first and later hardening.
